# Multi-ratings (BeachPlay + РПТТ) — план реализации

## Термины
- `BP`: BeachPlay рейтинг из `players_player.current_rating`.
- `РПТТ`: рейтинги из `btr_btrratingsnapshot` (датой снэпшота является `rating_date`).
- `rating_visible` (в турнире): какой рейтинг показывать/использовать для автопосева.

## 1) Схема данных турнира
- Добавить `tournaments_tournament.rating_visible` (choices):
  - `beachplay` (default)
  - `btr_mw` (РПТТ (м/ж))
  - `btr_mixed` (РПТТ (микст))
  - `btr_under` (РПТТ (юн.))
- Миграция.
- Сериализатор турнира: отдавать/принимать поле в create/update.

## 2) Источник и валидность РПТТ
- Для РПТТ брать **последний** snapshot, но только если `rating_date >= (start_of_current_month - 12 months)`.
- Если валидного снапшота нет: считать рейтинг/место отсутствующими (не показывать устаревшее).

## 3) Выбор РПТТ категории по полу
- Пол берём из `players_player.gender`, если null — из `btr_btrplayer.gender`, иначе null.
- Маппинг `rating_visible` -> категории:
  - `btr_mw`: M или W
  - `btr_mixed`: MX или WX
  - `btr_under`: MU или WU

## 4) Единый слой вычисления «видимого рейтинга/места»
Сделать единые функции (backend) и везде заменить текущие ветвления:
- `get_visible_player_rating(tournament, player, as_of_date)` -> `{place, value, label}`
- `get_visible_team_rating(tournament, team, as_of_date)` -> `{value, label}`
Где:
- Для BP: `value = player.current_rating`, `place` вычисляется как на странице `/rating`.
- Для РПТТ: `value/place` из snapshot по правилу 12 месяцев.

## 5) Отображение в UI (единый формат)
### 5.1 Одиночки
Всегда показывать рядом с игроком в скобках:
- `(#10 • 1263 BP)` или `(#22 • 863 РПТТ(м/ж))` и т.п.

### 5.2 Пары
- Если выбран режим показа **ФИО пары**:
  - рядом с парой показывать **одно** значение: team rating и суффикс `BP/РПТТ(...)`.
- Если выбран режим показа **имён по игрокам**:
  - для каждого игрока пары показывать его `(#place • rating label)`.

## 6) Team rating (пара) и автопосев
- Для BP: `team_rating = average(player1, player2)` (как сейчас).
- Для РПТТ: `team_rating = sum(player1, player2)`.
- Автопосев использовать `team_rating` согласно `tournament.rating_visible`.

## 7) Порядок игроков в паре (везде)
Правило сортировки игроков пары:
1) игрок с признаком `профи` — первым
2) если оба профи: женский пол первым (`gender=female`), `null` — вторым
3) если пол одинаковый (или оба null): выше рейтинг по `tournament.rating_visible` — первым
4) если всё равно равно: по алфавиту

## 8) Модалки турнира (frontend)
- В "Новый турнир" и "Изменить настройки" добавить поле "Отображать рейтинг" (radio) после "Рейтинг" и перед "Призовой фонд".
- Значения: `BeachPlay` (default), `РПТТ (м/ж)`, `РПТТ (микст)`, `РПТТ (юн.)`.

## 9) Глобальная замена текстов
- На фронте заменить все вхождения "BTR" -> "РПТТ" (только UI-текст, не обязательно internal codes).

## 10) Миграция игроков BTR -> BP
- Management command:
  - создать BP игроков для всех BTR игроков, которых ещё нет
  - связать `players_player.btr_player_id`
  - **для новых BP игроков** выставить `current_rating = 0`

## 11) Юниоры: стартовый рейтинг BP=800
- В существующей логике старта BP рейтинга:
  - если у BP игрока нет BP рейтинга, но есть `btr_player_id`, и есть **только** РПТТ(юн.) (MU/WU), то установить стартовый BP рейтинг `800`.

## Порядок реализации (рекомендуемый)
1) Инвентаризация мест использования рейтинга/места/посева.
2) `rating_visible` в модели турнира + UI модалок.
3) Backend слой видимого рейтинга/места + замена старых ветвлений.
4) Team rating (avg/sum) + автопосев.
5) Порядок игроков в паре.
6) Скрипт миграции BTR->BP.
7) Юниорский стартовый рейтинг.
